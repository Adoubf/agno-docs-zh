**Workspace** is an umbrella term for your application and its purpose is to help manage the software, application and infrastructure components as one codebase.

In other words, each application built with phidata is a **Workspace** that contains:

- **Software layer:** Conversations, agents, routes for serving them, django views, database tables, streamlit pages. This is code unique to your application.
- **Application layer:** The code for the applications that serve the software - FastApi, Streamlit, Django, PostgreSQL.
- **Infrastructure layer:** The code for the infrastructure that runs the applications - S3 buckets, RDS databases, ECS clusters and services.

## Create workspace

Phidata provides templates for common applications that you can use to create your workspace.

Run `phi ws create` to create a new workspace in the current directory. `phi` will ask for a workspace template and name if not provided.

<CodeGroup>

```bash Create Workspace
phi ws create
```

```bash Create LLM App
phi ws create -t llm-app -n llm-app
```

```bash Create Api App
phi ws create -t api-app -n api-app
```

```bash Create Web App
phi ws create -t django-app -n web-app
```

</CodeGroup>

## Start workspace

Run `phi ws up` to create resources defined in the `workspace`

<CodeGroup>

```bash Terminal
phi ws up
```

```bash Shorthand
phi ws up dev:docker
```

```bash With Options
phi ws up --env dev --infra docker
```

```bash Short Options
phi ws up -e dev -i docker
```

</CodeGroup>

## Command Options

<Note>Run `phi ws up --help` for all options</Note>

### Environment (`--env`)

Use the `--env` or `-e` flag to filter the environment

<CodeGroup>

```bash Flag
phi ws up --env dev
```

```bash Shorthand
phi ws up dev
```

```bash Short Options
phi ws up -e dev
```

</CodeGroup>

### Config (`--infra`)

Use the `--infra` or `-i` flag to filter the config

<CodeGroup>

```bash Flag
phi ws up --infra docker
```

```bash Shorthand
phi ws up :docker
```

```bash Short Options
phi ws up -i docker
```

</CodeGroup>

### Group (`--group`)

Use the `--group` or `-g` flag to filter the resource group.

<CodeGroup>

```bash Flag
phi ws up --group airflow-ws
```

```bash With Options
phi ws up \
  --env dev \
  --infra docker \
  --group airflow-ws
```

```bash Shorthand
phi ws up dev:docker:airflow-ws
```

```bash Short Options
phi ws up \
  -e dev \
  -i docker \
  -g airflow-ws
```

</CodeGroup>

### Name (`--name`)

Use the `--name` or `-n` flag to filter the resource name

<CodeGroup>

```bash Flag
phi ws up --name airflow-ws
```

```bash With Options
phi ws up \
  --env dev \
  --infra docker \
  --name airflow-ws
```

```bash Shorthand
phi ws up dev:docker::airflow-ws
```

```bash Short Options
phi ws up \
  -e dev \
  -i docker \
  -n airflow-ws
```

</CodeGroup>

### Type (`--type`)

Use the `--type` or `-t` flag to filter the resource type.

<CodeGroup>

```bash Flag
phi ws up --type container
```

```bash With Options
phi ws up \
  --env dev \
  --infra docker \
  --type container
```

```bash Shorthand
phi ws up dev:docker:airflow-ws::container
```

```bash Short Options
phi ws up \
  -e dev \
  -i docker \
  -t container
```

</CodeGroup>

### Dry Run (`--dry-run`)

The `--dry-run` or `-dr` flag can be used to **dry-run** the command. `phi ws up -dr` will only print resources, not create them.

<CodeGroup>

```bash Flag
phi ws up --dry-run
```

```bash With Options
phi ws up \
  --env dev \
  --infra docker \
  --dry-run
```

```bash Shorthand
phi ws up dev:docker -dr
```

```bash Short Options
phi ws up \
  -e dev \
  -i docker \
  -dr
```

</CodeGroup>
